# Plan JSON Schema

The plan is the colony's shared notepad. Agents both read and write it. It captures the evolving understanding of the gap between the goal and reality.

## Top-Level Structure

```json
{
  "project": "string - project name",
  "branchName": "string - git branch (e.g., colony/feature-name)",
  "goalFile": "string - relative path to goal document",
  "tasks": [ "...array of Task objects..." ],
  "metadata": { "...iteration tracking..." }
}
```

## Task Object

```json
{
  "id": "T-001",
  "title": "Short description of the work",
  "description": "Detailed description. Agents update this as they learn more.",
  "subtasks": [
    { "description": "Implementation step", "done": false }
  ],
  "votes": [
    {
      "iteration": 3,
      "role": "implementer",
      "value": 1,
      "reason": "Explanation of why this vote"
    }
  ],
  "comments": [
    {
      "iteration": 5,
      "role": "reviewer",
      "message": "Advisory nudge or observation"
    }
  ],
  "status": "open | in-progress | done",
  "createdBy": { "iteration": 1, "role": "implementer" },
  "claimedBy": { "iteration": 3, "role": "implementer" } | null,
  "splitFrom": "T-000" | null
}
```

## Field Details

### id
Sequential identifier: T-001, T-002, etc. Generated by `plan.sh create`.

### title
Short, imperative description. "Add priority field to database" not "The priority field should be added."

### description
Detailed context. Agents can and should update this as they learn more about what the task entails. This is a living document, not a frozen spec.

### subtasks
Implementation steps within the task. Agents add these during gap analysis when they see missing steps. Subtasks track completion (`done: true/false`) but do NOT have their own votes - voting stays at the task level to keep granularity right.

If a task accumulates too many subtasks (becoming unwieldy), agents should split it using `./plan.sh split`.

### votes
The primary trace mechanism. Each vote records:
- **iteration** - Which loop iteration cast this vote
- **role** - Which agent role (implementer/reviewer/simplifier)
- **value** - +1 (confirms task is needed) or -1 (challenges task)
- **reason** - Why this vote was cast (critical for future agents to understand)

Votes are never removed. Negative votes don't delete tasks - they serve as warnings and historical record.

### comments
Advisory notes, primarily from reviewers. These are nudges, not directives. Examples:
- "Reuse the existing Button component instead of creating a new one"
- "This depends on T-003 being finished first"
- "The error handling pattern in this codebase uses Result types, not try/catch"

### status
- **open** - Not yet started. Available for any implementer to claim.
- **in-progress** - Currently being worked on, or partially complete from a previous iteration.
- **done** - Fully implemented and verified.

Tasks are never deleted. Stale or unnecessary tasks accumulate -1 votes and remain as historical record.

### createdBy / claimedBy
Track which iteration and role created or claimed the task. Useful for understanding the history of a task.

### splitFrom
If this task was created by splitting a larger task, references the original task ID. Provides traceability.

## Metadata

```json
{
  "createdAt": "ISO 8601 timestamp",
  "totalIterations": 6,
  "iterationLog": [
    {
      "iteration": 1,
      "role": "implementer",
      "taskWorked": "T-001",
      "contextCompacted": false
    }
  ]
}
```

The iteration log tracks what happened each iteration - which role ran, what task was worked on, and whether context compaction occurred. This data helps identify patterns (e.g., frequent compaction = tasks too large).
